# Step 1：锁定 Dashboard 的“统计口径”与图表（最关键的设计锚点）

你已经确定：

- 默认指标集合：`acc, loss, f1`
- 趋势图粒度：`day`
- bestMetric 方向：跟随 `metric_def.direction`（MAX/MIN）

我们把 Dashboard 固定为 **3 图 + 1 TopN + 1 KPI 区**，并给每一块明确口径。

## 1.1 KPI 指标卡（Summary）

**输入过滤条件**：`projectId, dateFrom, dateTo, tagIds(可选), status(可选)`
**输出字段**：

1. `totalRuns`

- 口径：满足过滤条件的 run 总数

1. `runsLast7Days`

- 口径：在“当前过滤条件的 project + tag + status”下，再额外限定 `created_at >= today-7` 的 run 数

1. `successRate`

- 口径：`SUCCESS` / `totalRuns`（若 totalRuns=0，返回 0）

1. `bestMetric`（“最佳指标”）

- 口径：在过滤条件下，对某个“默认 metric”（建议用 `acc`）或前端选中的 metric：
  - 若 direction=MAX → 取最大 value
  - 若 direction=MIN → 取最小 value
- 输出：`metricDefId, metricName, value, runId`

> 这一块的“专业点”是：bestMetric 的方向来自 `metric_def.direction`，这非常像真实实验系统。

------

## 1.2 趋势图（Trend Line）

**目标**：展示某指标（acc/loss/f1）随时间变化
**输入**：`projectId, metricDefId, dateFrom, dateTo, tagIds, status, granularity=day`
**口径**（推荐最稳、最好讲清楚的口径）：

- x 轴：按天分桶（YYYY-MM-DD）
- y 轴：当天所有 run 的该指标的 **平均值 avg(value)**
  （也可选 max/min，但 avg 最通用）

> 你以后写文档时可以说：为了避免单次实验波动，我们用日聚合均值观察整体趋势。

------

## 1.3 分布图（Distribution）

**目标**：按某维度统计实验数量分布
**输入**：`projectId, by=tag|model|dataset, dateFrom, dateTo, status`
**口径**：count(runs) 聚合

- by=model：`run.model_name` 计数
- by=dataset：`run.dataset_name` 计数
- by=tag：通过 `run_tag` 连接统计

> 这块非常“可视化友好”，演示时也很直观。

------

## 1.4 TopN（排行榜）

**目标**：展示最好的 N 个实验记录
**输入**：`projectId, metricDefId, order(由 direction 决定), limit=10, dateFrom, dateTo, status`
**口径**：按 `run_metric.value` 排序，取前 N

输出字段（给前端表格用）：

- runId, runName, modelName, datasetName, value, createdAt, status

------

✅ 到这里，Dashboard 的“数学定义”已经固定。
**下一步**就是把这些口径落成“数据库查询设计”。

# Step 2：把口径翻译成“数据库查询/SQL 设计”（你后端实现的蓝图）

下面我用“接近 SQL 的形式”描述，每一条都能直接翻译成 MyBatis-Plus 或 XML SQL。

> 说明：这里我用 `run`、`run_metric`、`metric_def`、`run_tag` 这几张表。
> 过滤条件写成 `WHERE ...`，你后面实现时就是拼 QueryWrapper / 自定义 SQL。

------

## 2.1 Summary：totalRuns、successRate、runsLast7Days

### (a) totalRuns

```sql
SELECT COUNT(*) AS totalRuns
FROM run r
WHERE r.project_id = :projectId
  AND r.created_at BETWEEN :dateFrom AND :dateTo
  AND (:status IS NULL OR r.status = :status)
  AND (:tagFilterOff OR EXISTS (
      SELECT 1 FROM run_tag rt
      WHERE rt.run_id = r.id AND rt.tag_id IN (:tagIds)
  ));
```

### (b) successRate

successRate 实现建议：一次性查两列，避免查两次：

```sql
SELECT
  COUNT(*) AS totalRuns,
  SUM(CASE WHEN r.status='SUCCESS' THEN 1 ELSE 0 END) AS successRuns
FROM run r
WHERE ...同上过滤...
```

后端算：`successRate = successRuns / totalRuns`（注意 totalRuns=0）

### (c) runsLast7Days

你可以复用过滤条件，但时间窗变为最近7天：

```sql
SELECT COUNT(*) AS runsLast7Days
FROM run r
WHERE r.project_id = :projectId
  AND r.created_at >= :todayMinus7
  AND ...status/tag过滤同上...
```

------

## 2.2 Summary：bestMetric（自动 MAX/MIN）

bestMetric 的关键是 **direction**。

### 步骤 1：先取 direction

```sql
SELECT direction, name
FROM metric_def
WHERE id = :metricDefId;
```

### 步骤 2：按 direction 排序取第一名

```sql
SELECT rm.run_id, rm.value
FROM run_metric rm
JOIN run r ON r.id = rm.run_id
WHERE r.project_id = :projectId
  AND rm.metric_def_id = :metricDefId
  AND r.created_at BETWEEN :dateFrom AND :dateTo
  AND (:status IS NULL OR r.status = :status)
  AND (:tagFilterOff OR EXISTS (...run_tag...))
ORDER BY rm.value DESC  -- 若 direction=MAX
LIMIT 1;
```

若 direction=MIN，就改成 `ORDER BY rm.value ASC`

输出再补充 `metricName`（来自 metric_def）即可。

------

## 2.3 Trend：按天聚合 avg(value)

```sql
SELECT
  DATE(r.created_at) AS x,
  AVG(rm.value) AS y
FROM run_metric rm
JOIN run r ON r.id = rm.run_id
WHERE r.project_id = :projectId
  AND rm.metric_def_id = :metricDefId
  AND r.created_at BETWEEN :dateFrom AND :dateTo
  AND (:status IS NULL OR r.status = :status)
  AND (:tagFilterOff OR EXISTS (...run_tag...))
GROUP BY DATE(r.created_at)
ORDER BY x ASC;
```

> 注意：如果某天没有数据，前端可以不显示，或者后端补零（后端补齐会更复杂，建议前端简单处理）。

------

## 2.4 Distribution：按 model/dataset/tag

### by=model

```sql
SELECT r.model_name AS name, COUNT(*) AS value
FROM run r
WHERE r.project_id = :projectId
  AND r.created_at BETWEEN :dateFrom AND :dateTo
  AND (:status IS NULL OR r.status = :status)
GROUP BY r.model_name
ORDER BY value DESC;
```

### by=dataset 同理

### by=tag

```sql
SELECT t.name AS name, COUNT(*) AS value
FROM run r
JOIN run_tag rt ON rt.run_id = r.id
JOIN tag t ON t.id = rt.tag_id
WHERE r.project_id = :projectId
  AND r.created_at BETWEEN :dateFrom AND :dateTo
  AND (:status IS NULL OR r.status = :status)
GROUP BY t.id, t.name
ORDER BY value DESC;
```

------

## 2.5 TopN：按 metric 排序取前10

```sql
SELECT
  r.id AS runId,
  r.run_name AS runName,
  r.model_name AS modelName,
  r.dataset_name AS datasetName,
  rm.value AS value,
  r.created_at AS createdAt,
  r.status AS status
FROM run_metric rm
JOIN run r ON r.id = rm.run_id
WHERE r.project_id = :projectId
  AND rm.metric_def_id = :metricDefId
  AND r.created_at BETWEEN :dateFrom AND :dateTo
  AND (:status IS NULL OR r.status = :status)
ORDER BY rm.value DESC   -- 由 direction 决定
LIMIT :limit;
```

# Step 3：实现策略（用 MyBatis-Plus 最稳的做法）

你用 MyBatis-Plus 的时候，最稳的原则是：

- **CRUD 用 MP（省时）**
- **聚合/图表查询用自定义 Mapper SQL（XML 或 @Select）**
  因为 MP 的 QueryWrapper 做复杂 group by / join 会变得很难读，也更容易 AI 写错。

所以分工是：

## 3.1 MP 管 CRUD（Runs/Projects/Tags/Metrics）

- `ProjectService`：分页查询、CRUD
- `RunService`：分页查询、CRUD（涉及 run_tag & run_metric 用事务）
- `MetricDefService`：指标字典 CRUD
- `TagService`：标签 CRUD

## 3.2 自定义 DashboardMapper（专门做聚合查询）

建一个 `DashboardMapper`，里面写：

- `selectSummaryCounts(...)`
- `selectBestMetric(...)`
- `selectTrend(...)`
- `selectDistributionByModel(...)`
- `selectDistributionByDataset(...)`
- `selectDistributionByTag(...)`
- `selectTopRuns(...)`

这样代码结构非常“工程化”，写文档也好写：

> “CRUD 与统计查询解耦；统计查询集中在 Dashboard 模块，便于优化。”

# Step 6：更新趋势图口径（T2：按天 best）

我们把 Trend 从“avg(value)”改为“按天取 best”，best 的方向取决于 `metric_def.direction`：

- direction=MAX（acc/f1）→ 当天 y = MAX(value)
- direction=MIN（loss）→ 当天 y = MIN(value)

## 6.1 Trend SQL（T2）

**先查 direction**（同前）：

```sql
SELECT direction FROM metric_def WHERE id=:metricDefId;
```

然后按天聚合：

```sql
SELECT
  DATE(r.created_at) AS x,
  /* direction=MAX */ MAX(rm.value) AS y
FROM run_metric rm
JOIN run r ON r.id = rm.run_id
WHERE r.project_id = :projectId
  AND rm.metric_def_id = :metricDefId
  AND r.created_at BETWEEN :dateFrom AND :dateTo
  AND (:status IS NULL OR r.status = :status)
  AND (:tagFilterOff OR EXISTS (...run_tag...))
GROUP BY DATE(r.created_at)
ORDER BY x ASC;
```

若 direction=MIN，把 `MAX` 换成 `MIN`。

> 文档可写：趋势图展示“每日最佳实验结果（Best-of-day）”，更贴近深度学习实验对比习惯。

## 7.2 Metrics 动态表单（核心）

UI 结构建议：表格形态最清晰

- 用一个数组 `metrics: Array<{ metricDefId: number; value: number }>` 作为表单字段
- 页面上展示一个 `<el-table :data="metrics">`
- 每行两列：
  - 指标：`<el-select>`（从 `/api/metrics/defs` 拉；显示 display_name）
  - 值：`<el-input-number>`
- 行操作：
  - “添加指标”按钮：push 一行空的
  - “删除”按钮：remove 该行
- 校验规则：
  - metricDefId 必填
  - value 必填
  - 不允许重复 metricDefId（前端校验：Set 判重，报错提示）

> 这个动态表单是你“工作量与技术复杂度”的可见亮点：老师看到就会觉得你们做得像真系统。

# Step 10：后端事务边界（让 AI 写代码也不容易坏）

你选择了多表（run + run_metric + run_tag），所以创建/更新必须是事务：

### Run 创建（事务内）

1. insert run
2. insert run_metric（批量）
3. insert run_tag（批量）

### Run 更新（事务内）

1. update run
2. delete old run_metric where run_id=?
3. insert new run_metric
4. delete old run_tag where run_id=?
5. insert new run_tag

> 这块写进文档就是“工程性”的体现。
